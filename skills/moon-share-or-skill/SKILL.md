# moon-share-or-skill

把"刚做完一件事"的经验,沉淀到正确的去处:

- **moon-share** (`D:/moon-ai-work/moon-share/`) — 对外内容,公众号/小红书/朋友圈/视频号/抖音等渠道
- **moon-skills** (`D:/moon-ai-work/moon-skills/`) — 对内能力,给 Claude Code / Codex / Cursor 等 agent 复用的 skill

或者:判定**不需要沉淀**。并不是所有经验都值得留下 —— 过度沉淀和不沉淀一样糟。

这个 skill 的目标是**帮用户做判断**,而不是立即开写。输出**是一个判断报告**,不是一篇文章或一份 skill。

## 何时触发

**用户显式调用时** —— 类似下面这些说法都要触发:
- "沉淀一下这次经验"
- "把这次的经验整理成 skill"
- "这个能做成 skill 吗"
- "做个总结"
- "/moon-share-or-skill"

**主动提议时机** —— 当你(Claude)观察到以下情形,应当**问一次**"要不要用 moon-share-or-skill 把这次经验沉淀一下?":
- 刚写完一份有分量的 spec / design / plan 文档(尤其在 `docs/superpowers/specs/` 或 `docs/superpowers/plans/` 下)
- 刚完成一个跨越多次会话 / 多个阶段的任务
- 用户对刚走过的流程表达了"学到东西了"/"这个方法挺好的"之类的反馈
- 用户即将结束当前任务切去做别的,而本次经验还没有被沉淀

**主动提议时不强推**。如果用户说"不用",就不再提。每次会话最多主动问一次。

## 两种执行模式

开始执行时,**先问用户选哪一种**:

| 模式 | 说明 | 适用场景 |
|---|---|---|
| **新会话模式** | 生成一段可复制到新会话的完整提示,用户在新会话里执行沉淀工作 | 当前会话上下文已经塞得较满,或沉淀需要大块独立空间;推荐默认 |
| **当场模式** | 在当前会话里直接读文档、给判断、(如用户同意)直接写入 moon-share / moon-skills | 当前会话干净、手头材料都在、想要一气呵成 |

问法示例:

> 要走哪种模式?
> - **新会话模式**(推荐):我生成一段提示,你复制到新会话执行。不占用当前上下文。
> - **当场模式**:我直接读文档给你做判断,你同意后我直接动手。

## 标准流程(两种模式共用的骨架)

无论哪种模式,沉淀判断都按这个骨架走:

### 步骤 1 · 识别输入材料

确认**这次要沉淀的"经验"具体是哪些文档 / 哪段对话 / 哪个 commit 范围**。

- 如果用户只说"这次经验",要追问:具体是指哪次?哪些文件?
- 常见输入:`docs/superpowers/specs/` 下的 spec 文件、`docs/superpowers/plans/` 下的 plan 文件、最近几个 commit、本次会话的对话纪录
- **至少要有明确的文档路径或 commit 范围**,否则判断没有依据

### 步骤 2 · 通读并抽取"值得留下的东西"

读完材料,抽取以下几类信号:

1. **方法论 / 流程**(可以被别人复用的做事方法)—— 往 skill 方向
2. **可教学的案例 / 决策过程**(有故事、有取舍)—— 往 share 方向
3. **技术决策 / 工具组合 / 踩坑**(带反直觉的结论)—— share 或 skill 都可能,看表达形式
4. **一次性的、与当前具体项目强绑定的细节** —— 不沉淀,留在项目文档里

### 步骤 3 · 给三分类判断报告

输出格式如下(不要省略任何一节,即使某一节为空):

```
## 沉淀判断

### 🔧 建议抽成 moon-skills 的
- [skill 名字候选]
  - 核心能力: 一句话
  - 触发场景: 什么时候该用
  - 来源材料: <文档路径 或 对话段落>
  - 理由: 为什么值得做成 skill 而不是写成文章

### 📝 建议写成 moon-share 内容的
- [文章标题候选]
  - 渠道: 公众号 / 小红书 / 视频号 等(选 1-2 个最合适的)
  - 核心观点: 一句话
  - 素材点: 从原材料里哪几段可以直接用
  - 理由: 为什么适合对外分享而不是沉淀为 skill

### 🗑️ 不建议沉淀的
- [要点]
  - 为什么不沉淀: 一次性 / 项目强绑定 / 已有更好来源 / 过于琐碎
```

**不要空泛。每一项都要带具体文件路径 / 对话引用 / commit hash。**

### 步骤 4 · 等用户确认取舍

用户看完判断,会选择:

- 哪些条目要真的落地(其余忽略)
- 每个落地条目的优先级

**此时严禁直接开写。** 让用户先挑。

### 步骤 5 · 按用户选择落地(仅当场模式会到这一步)

- **skill 落地**:在 `D:/moon-ai-work/moon-skills/skills/<skill-name>/SKILL.md` 下写文件,遵循本 skill 自身的格式(frontmatter 只写"何时使用")
- **share 落地**:在 `D:/moon-ai-work/moon-share/` 下按渠道分目录(公众号/小红书/视频号/抖音)建 markdown 文件。文件名 `YYYY-MM-DD-<slug>.md`
- **全部落地后询问**是否 `git commit`,以及是否 `git push` —— 默认只 commit 不 push

新会话模式下,这一步交给用户在新会话里完成,当前 skill 不直接动文件。

## 新会话模式的输出模板

当用户选"新会话模式",输出格式如下(直接给用户,不套别的话):

> ---
>
> **开一个新会话**,把下面这段完整发出去(保留代码块格式):
>
> \`\`\`
> 这次会话不写 <当前项目名> 的代码,而是做经验沉淀。
>
> 请按 moon-skills 的 `moon-share-or-skill` skill 执行。
>
> 这次要沉淀的材料:
> - <文档路径 1,一行一条>
> - <文档路径 2>
> - <commit 范围 或 对话要点>
>
> 候选沉淀去处:
> - D:/moon-ai-work/moon-share/  (对外内容)
> - D:/moon-ai-work/moon-skills/ (对内能力)
>
> 先读材料,按 skill 的"标准流程"输出三分类判断报告(🔧 / 📝 / 🗑️)。
> 不要直接开写,等我挑完再说。
> \`\`\`
>
> ---

**生成这段时要把 `<...>` 占位符全部换成真实路径**,不能留尖括号。

## 判断准则(防止拍脑袋)

这些是**用来否决"显然值得沉淀"的直觉**的检查:

### 判断一条经验是否该做 skill

- [ ] 它是一种**方法或流程**(不是一次性的事实)
- [ ] 它**跨项目可复用**(不只适用于当前一个仓库)
- [ ] 它**不是已有 superpowers skill 能覆盖的**(比如不要做"规划""TDD""调试"这种已经存在的)
- [ ] 它**有清晰的触发条件**(能写进 frontmatter 的 "Use when ...")
- [ ] 它的核心价值是"让 agent 不犯某类错"而不是"给 agent 更多背景"(**背景信息应该进 CLAUDE.md,不是 skill**)

**5 条全过才做 skill。** 任何一条不过都按"不沉淀"或"改做 share"处理。

### 判断一条经验是否该做 share 内容

- [ ] 它有**一个清晰的主旨**(一句话能说完)
- [ ] 它**对外部读者**有信息密度,不只是"我做了什么"
- [ ] 它**带反直觉的点 / 具体数字 / 可复制的模板**中至少一种
- [ ] 你有办法用**一个合适的渠道形态**呈现(公众号长文 / 小红书卡片 / 视频号脚本 等)

**4 条全过才做 share。** 否则放弃。

### 反模式(以下情况直接归入"不沉淀")

- 只是一个 bug 修复记录 —— 在 commit message 里已经够了
- 只是"我学会了 X 工具的用法" —— 官方文档是更好来源
- 内容高度依赖当前项目的特定结构 —— 属于 CLAUDE.md,不是 skill
- 一条感慨("这次 AI 真厉害") —— 不是可传递的信息
- 对现有 skill 的轻微改写 —— 应该**改那个 skill**,而不是新建一个

## 迭代准则

- **宁缺毋滥**。moon-skills 里 10 个低质量 skill 比 3 个高质量 skill 糟糕得多 —— 前者会让用户对整个仓库失去信任。
- **一次沉淀不要产出超过 2 个 skill** 或 **超过 2 篇 share**。如果材料里真有更多值得留的东西,也要排优先级,第一次只做最有价值的。
- **skill 命名用小写连字符**:`moon-share-or-skill`、`ri-pao-platform-design`,不要驼峰、不要中文、不要空格。
- **share 命名用日期前缀**:`2026-04-23-ri-pao-platform-design-brainstorm.md`,便于按时间排序。

## 输出纪律

- 判断报告要**精确到文件路径和章节**,不要笼统地说"这次 brainstorm 经验"
- 每条候选 skill / share 都要给出**具体的理由**,而不是"这个很值得沉淀"
- 执行 "当场模式" 写入文件前,**必须等用户明确说"同意 / 按这个做"**,不能基于"看起来用户没反对"就动手
- 写完文件后,**用 Read 工具读回确认**,避免 Write 成功但内容有问题
